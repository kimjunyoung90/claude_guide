# Claude 교육 PPT 작업 지침서

PPT 제작 과정에서 내린 결정사항과 작업 원칙을 정리한 문서.

---

## 1. 작업 프로세스

### 반드시 지킬 것
- **내용 수정 전 반드시 방향 검증** — "앞으로 ppt 수정할 때 나랑 내용 및 방향 검증하고 해"
- **slides.md, script.md, HTML은 병렬적으로 수정** — 하나만 수정하고 나머지를 빠뜨리지 말 것
- **공식 문서 기반** — 내용의 정확성은 Anthropic 공식 문서로 검증할 것

### 하지 말 것
- 검증 없이 임의로 내용 추가/수정하지 않기
- 불확실한 정보를 넣지 않기 (확인 안 된 내용은 반드시 공식문서 확인)

## 2. 교육 대상 및 맥락

| 항목 | 내용 |
|------|------|
| 대상 | 사내 개발자, QA, 퍼블리셔 (한 세션에 함께) |
| 시간 | 20~30분 |
| 목적 | Claude Code(CLI)를 실무에 쓰자 |
| 기대 결과물 | 각 팀이 자기 업무에 클로드를 적용한 사례 공유 |
| 마무리 | 데모 영상으로 실무 워크플로우를 보여주면서 마무리 |

### 대상별 고려사항
- 개발자만 아는 용어 사용 주의 (QA, 퍼블리셔도 이해해야 함)
- "프로젝트"라는 표현은 QA에게 안 와닿을 수 있음 → 직군별 예시를 나눠서 제공 (개발팀 + QA팀)
- `package.json` 같은 개발 특화 파일이 보이면 QA가 와닿지 않음

## 4. 용어 및 표현 결정사항

### 토큰 (Token)
- **확정 표현**: "영어는 짧은 단어 하나가 대략 1토큰, 한글은 같은 의미라도 더 많은 토큰 소모"
- "영어 3.5글자" → 무슨 말인지 모르겠음 → "짧은 단어 하나"로 변경
- "Hello = 1토큰" → 부정확 → 수정함
- 공식문서 기준: 1 token ≈ 3.5 English characters

### 컨텍스트 (Context)
- **확정 표현**: "AI가 답변 생성 시 참고할 수 있는 텍스트의 양"
- 컨텍스트에 포함되는 것: 대화 내용 + 파일 내용 + 설정 = 전부
- AI가 읽어들인 모든 정보(파일, 검색, 도구 결과)는 컨텍스트에 포함됨
- **미결**: "기억" vs "소화" — 어떤 단어가 더 적절한지 확정 필요
- 컨텍스트가 가득 차면 → "대화 내용을 잊어버리거나, 중요한 맥락을 못 짚는 경우 발생"

### 컨텍스트 윈도우 크기
- Opus/Sonnet: 100만 토큰 (≈ 75만 단어) — 구체적 수치로 와닿게
- Haiku: 20만 토큰
- "1M 토큰"이라는 표기는 안 넣어도 됨

### CLAUDE.md
- **확정 표현**: "Claude가 자동으로 읽는 기본 지침서"
- "우리 팀 규칙서" → 안 와닿음 → 변경
- 200줄 이하 권고

### Skills
- **확정 표현**: "자주 쓰는 작업을 /명령어로 등록"
- SKILL.md 500줄 이하 권고, 초과 시 별도 파일로 분리
- 호출: name 필드가 있으면 name으로, 없으면 폴더명으로 호출
- description 필드를 보고 자동 호출 판단

### MCP
- **확정 표현**: "Claude에 외부 프로그램을 연결해주는 도구"
- "외부 서비스 연결 통로" → 와닿을까 의문 → "연결해주는 도구"로 확정
- "프로그램"이 2번 나오는 건 어색하지만 수용
- **핵심 메시지**: "MCP는 토큰 먹는 귀신" — 대화 시작 시 활성화된 모든 MCP의 사용법을 컨텍스트에 포함시킴
- MCP 설명 슬라이드에서 위 내용 강조, 컨텍스트 관리 원칙에서는 "미사용 MCP 비활성화"만 간략히

### Claude를 만나는 방법 (Slide 2)
- "어디서 쓸 수 있나" → "어떻게 쓸 수 있나" → "여러가지 방식으로 제공한다" → 전부 어색 → **"Claude를 만나는 방법"**으로 확정
- 각 방식의 차이: 접근성 차이 + 유/무료 차이

### Subagents
- "별도 컨텍스트에서 일하는 독립 작업자, 결과만 반환"
- Skills과의 차이: Subagents는 별도 컨텍스트에서 작업 → 메인 컨텍스트 오염 방지
- 예시: "Confluence에서 결제 서비스 정책 찾아줘" → 직접 하면 탐색 내용이 컨텍스트를 차지, Subagent에게 위임하면 결과만 받음

### Agent Teams
- Subagents와의 차이: **의존적 작업에서 팀원끼리 직접 메시지를 주고받으며 협업**
- 독립적 작업 → Subagents, 의존적 작업 → Agent Teams
- 프롬프트: "에이전트 팀을 구성해줘"가 아니라 "팀을 구성해서 처리해줘" 뉘앙스

### Hooks, Plugins, /loop
- **간단히만** 다룸 ("Hooks + Plugins + /loop만 간단히")
- Hooks: 이벤트 발생 시 자동 실행 스크립트. settings.json에 조건, hooks/ 폴더에 스크립트
- Plugins: 기능들을 묶어서 공유하는 패키지 ("팀끼리"라는 단어는 빼기)
- /loop: 세션 안에서 프롬프트 반복 실행 (세션 닫으면 끝)
- Cron도 세션 스코프 (OS 레벨이 아님)

## 5. 컨텐츠 구성 원칙

### 슬라이드 순서 (확정)
1. Claude란? (모델 종류 + 할 수 있는 것들)
2. Claude를 만나는 방법 (웹/Desktop/Code 비교)
3. 알아두면 좋은 기본 개념 (토큰, 컨텍스트)
4. 왜 불필요한 정보가 성능을 떨어뜨리는가?
5. Claude Code의 핵심 기능들 (전체 폴더 구조 + 기능 목록)
6. CLAUDE.md
7. Skills
8. MCP
9. Subagents
10. Agent Teams
11. 기타 기능 (Hooks, Plugins, /loop)
12. 컨텍스트 관리 원칙
13. 데모

### 넣은 것
- 각 슬라이드에 해당 공식 문서 링크 (우하단, 빨간색, 클릭 가능)
- 개발팀 + QA팀 두 가지 예시 (CLAUDE.md 등)
- 터미널 UI 재현 (커서 깜빡임, /doctor 모달, skill 호출 데모)
- 컨텍스트 관리 원칙에 /doctor 모달 팝업

### 뺀 것
- 토큰 비용/과금 관련 내용 전부 → "전달하고 싶은 건 컨텍스트가 가득 찰수록 성능 저하된다는 것"
- 5분 실습
- 다음 단계 안내
- 데모 포인트 정리
- "개발/QA/퍼블리싱 팀 대상"이라는 문구
- "왜 Claude Code인가?" 섹션
- 클로드 모델 종류 상세 (간단히만)
- 연구가 증명한 컨텍스트 관리 내용
- "컨텍스트 = AI의 작업 책상" 비유
- orders 테이블 조회 프롬프트 예시

## 6. 디자인/시각화 원칙

- 컨텐츠는 **크게크게** — 잘 보이도록
- 시각화는 **단순 명확하게** — 토큰 비교는 숫자(3 vs 8), 컨텍스트는 3단계 색상 게이지
- 터미널 UI를 HTML로 재현 (깜빡이는 커서, MCP 연결 상태 등)
- 인터랙션 요소: 클릭 가능한 모달 (doctor, SKILL.md 분리 가이드)
- 폴더 구조는 같은 슬라이드 안에서 보여주기 ("이런 식으로 폴더가 구성된다")
- 공식문서 링크: 우하단, 빨간색 테두리, z-index:10으로 클릭 가능하게

## 8. 미결 사항

1. **"기억" vs "소화"** — 컨텍스트 정의에서 어떤 단어가 적절한지 미확정
2. **Skills 폴더 구조의 examples/** — 제거할지 유지할지 미결정
3. **slides.md / script.md 동기화** — HTML과 아직 싱크 안 맞음
