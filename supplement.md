## 3. 핵심 단축키

### Escape — 작업 중지 / 되돌리기

| 동작 | 설명 |
|------|------|
| **1번 누르기** | 현재 진행 중인 작업을 중지한다. |
| **2번 누르기** | **rewind/summarize** 다이얼로그가 열린다. 코드와 대화를 이전 시점으로 복원하거나, 특정 메시지부터 요약할 수 있다. |

### 기타 단축키

| 단축키 | 설명 |
|--------|------|
| 스크린샷 붙여넣기 | 터미널에 스크린샷을 직접 추가할 수 있다. |
| `!` + 명령어 | Claude Code 안에서 bash 명령어를 실행할 수 있다. (예: `!git status`) |

### 음성 입력 모드

키보드 대신 **음성으로 프롬프트를 입력**할 수 있다.

- **활성화**: `Option + V` (Mac) 또는 설정에서 `"voiceEnabled": true`
- **사용법**: 단축키를 누르고 말한 뒤 다시 누르면 음성이 텍스트로 변환되어 입력된다.
- 긴 지시사항을 타이핑하기 번거로울 때 유용하다.
- 한국어 음성 인식을 지원한다.

### 세션 관리

| 명령어 | 설명 |
|--------|------|
| `/resume` | 이전 세션으로 복귀한다. `claude --resume`으로도 동일하게 사용 가능. |
| `/export` | 현재 대화를 텍스트 파일로 내보내거나 클립보드에 복사한다. |
| `claude --continue` | 방금 종료된 대화를 재시작한다. |

### 세션 기록 — Claude Code가 대화를 저장하는 방식

Claude Code는 대화 내역을 **실시간으로** 로컬 파일에 기록한다. 종료 시 한번에 저장하는 것이 아니라, 메시지가 오갈 때마다 즉시 파일에 추가(append)한다. 따라서 대화 도중 터미널이 갑자기 꺼져도 그 시점까지의 기록은 남아있고, `/resume`으로 복원할 수 있다.

**저장 위치:** `~/.claude/projects/<프로젝트경로>/<session-id>.jsonl`

**저장 내용:** 사용자 메시지, Claude 응답, 도구 호출, 도구 결과, 에러 등 **모든 상호작용**이 JSONL 형식으로 기록된다.

**자동 정리:** 마지막 활동 후 **30일이 지난 세션은 자동 삭제**된다. `settings.json`에서 변경 가능:
```json
{ "cleanupPeriodDays": 60 }
```

**중요:** 30일이 지나면 세션 기록이 삭제되어 `/resume`으로 복원할 수 없다. 보존이 필요한 대화는 **30일 안에 `/export`로 내보내둬야 한다.**

## 8. LLM의 동작 방식 — "기억"이 아니라 "확률 계산"

### Claude는 아무것도 기억하지 않는다

Claude는 대화가 끝나면 기억이 사라지는 게 아니라, **애초에 기억하는 것 자체가 없다.** LLM은 주어진 텍스트를 읽고, 다음에 올 가장 적절한 토큰을 확률적으로 예측하는 모델이다. 매번 입력을 받아서 확률 계산을 할 뿐, 뭔가를 "알고 있는" 상태가 아니다.

### 세션 중에도 "기억"이 아니다

세션 중 대화가 이어지는 것처럼 보이지만, 실제로는 **매번 전체 대화 내역을 입력으로 받아서 처음부터 읽는 것**이다. 대화가 쌓일수록 입력이 점점 길어지고, 컨텍스트 윈도우 한계에 도달하면 오래된 내용이 압축되거나 잘린다.

### Prompt Caching — 계산 효율화

사용자는 매번 전체 대화 내역을 Claude에게 전달하지만, Claude는 이미 계산한 부분을 **캐시(KV Cache)**로 관리하여 재계산하지 않는다. 이를 통해 속도와 비용을 절감한다.

- 캐시 히트 시 **비용 90% 절감, 속도 최대 85% 향상**
- 캐시는 마지막 사용 후 **5분이 지나면 만료** (사용할 때마다 갱신)
- 캐시는 계산 효율의 문제이고, 컨텍스트 윈도우 한계와는 별개

### 대화 저장은 모델이 아니라 도구가 한다

"세션이 끝나면 대화가 휘발된다"는 표현은 정확하지 않다. Claude 모델은 항상 무상태(stateless)이고, 대화 저장 여부는 **어떤 도구/서비스를 사용하느냐**에 달려 있다.

| | 대화 저장 | 누가 저장 |
|--|----------|----------|
| **Claude API (직접)** | 직접 관리해야 함 | 개발자 |
| **웹 Claude / Desktop** | 자동 저장 | 서비스(Anthropic) |
| **Claude Code** | 자동 저장 | Claude Code (로컬 파일) |

Claude Code는 대화를 실시간으로 로컬 파일에 기록하므로, 세션이 종료되어도 `/resume`으로 이전 대화를 불러올 수 있다. 하지만 이건 **Claude Code가 제공하는 기능**이지, Claude 모델이 기억하는 것이 아니다.

## PPT 보충 예정 내용

### 슬라이드 04 (컨텍스트가 길어지면 무슨 일이 생기는가)

- **Lost in the Middle 현상** 추가: 컨텍스트의 처음과 끝은 잘 기억하지만, 중간에 있는 정보는 놓치는 현상이 발생한다. (Stanford, Liu et al., 2023)
- **프롬프트 엔지니어링 vs 컨텍스트 엔지니어링**: 프롬프트 엔지니어링은 "어떻게 질문할까"에 집중하지만, 컨텍스트 엔지니어링은 "어떤 정보를 넣어줄까"에 집중한다. AI의 성능은 질문을 잘 하는 것보다 적절한 맥락을 제공하는 것에 더 크게 좌우된다.
- **컨텍스트의 종류**:
  - **System Context** — 시스템 프롬프트, CLAUDE.md, rules 등 AI의 행동 방식을 정의하는 지침
  - **Conversation Context** — 사용자와 AI 간의 대화 히스토리. 대화가 길어질수록 누적되어 컨텍스트 윈도우를 차지
  - **Tool & Environment Context** — 파일 읽기, 검색 결과, MCP 도구 정의 등 외부에서 주입되는 정보

### Hooks

Claude Code는 특정 이벤트 발생 시 자동으로 실행되는 **Hooks**를 지원한다. 4가지 유형으로 분류된다.

| 유형 | 설명 | 예시 |
|------|------|------|
| **Pre-action** | 작업이 실행되기 전에 실행 | SessionStart, PreToolUse, PreCompact |
| **Post-action** | 작업이 완료된 후 실행 | SessionEnd, PostToolUse, PostCompact, Stop |
| **Validation** | 작업을 허용/차단/수정할 수 있음 | UserPromptSubmit, PermissionRequest, TaskCompleted |
| **Notification** | 이벤트를 관찰하고 반응 (차단 불가) | InstructionsLoaded, SubagentStart, Notification |

### Subagents 모델 선택

Subagent는 별도 컨텍스트에서 독립적으로 작업하는 만큼, 반드시 Opus를 쓸 필요가 없다. 작업 성격에 맞는 모델을 선택하면 토큰을 효율적으로 사용할 수 있다.

| 작업 성격 | 추천 모델 | 이유 |
|-----------|----------|------|
| 파일 탐색, 단순 검색 | **Haiku** | 빠르고 저렴. 단순 조회에 충분 |
| 코드 리뷰, 요약, 분석 | **Sonnet** | 판단력과 비용의 균형 |
| 복잡한 설계, 아키텍처 결정 | **Opus** | 높은 추론 능력이 필요한 경우에만 |

커스텀 에이전트의 frontmatter에서 `model` 필드로 지정한다:

```markdown
---
name: code-searcher
model: haiku
---
```

API 기준으로 Opus 대비 Sonnet은 Input 40%·Output 40%, Haiku는 Input 20%·Output 20% 수준이므로 적절한 모델 선택만으로도 비용을 크게 줄일 수 있다.

### Subagents 병렬 실행

독립적인 작업은 Subagent를 **동시에 여러 개 실행**하여 속도를 높일 수 있다.

**프롬프트 예시:**

```
이 프로젝트를 병렬로 분석해줘:
1. 보안 취약점 점검
2. 성능 병목 분석
3. 테스트 커버리지 확인
```

이렇게 요청하면 Claude가 3개의 Subagent를 동시에 띄워 각각 독립된 컨텍스트에서 작업한 뒤, 결과만 모아서 보고한다.

**순차 실행 vs 병렬 실행:**

```
[순차] 보안 점검(2분) → 성능 분석(2분) → 테스트 확인(2분) = 총 6분
[병렬] 보안 점검(2분) ┐
       성능 분석(2분) ├→ 총 2분
       테스트 확인(2분) ┘
```

**언제 병렬로 쓰나:**

| 상황 | 병렬 적합 여부 |
|------|--------------|
| 서로 독립적인 분석 작업 | 적합 |
| 여러 파일을 각각 리뷰 | 적합 |
| 앞 작업 결과가 다음에 필요 | 부적합 (순차로) |

### 멀티 인스턴스 운영

터미널 탭을 여러 개 열어 Claude Code를 동시에 실행할 수 있다. 각 인스턴스는 독립된 세션으로 동작하며, 같은 프로젝트에서도 서로 다른 작업을 병렬로 진행할 수 있다.

**활용 예시:**

| 탭 | 작업 |
|----|------|
| 터미널 1 | 기능 개발 (feature 브랜치) |
| 터미널 2 | 버그 수정 (hotfix 브랜치) |
| 터미널 3 | 테스트 케이스 작성 |

**주의사항:**

- 같은 파일을 동시에 수정하면 **충돌이 발생**할 수 있다. 브랜치를 나눠서 작업하는 것을 권장한다.
- 각 인스턴스는 별도 컨텍스트를 사용하므로 **토큰이 각각 소비**된다.
- `--resume`이나 `/resume`으로 특정 세션을 골라 복귀할 수 있다.

### Auto Memory — Claude가 자동으로 학습하는 기억 시스템

CLAUDE.md는 사용자가 직접 작성하는 지침서이지만, Auto Memory는 **Claude가 대화에서 스스로 학습하여 자동 저장**하는 기능이다.

| | 프로젝트 CLAUDE.md | `~/.claude/CLAUDE.md` | Auto Memory |
|--|-----------|-------------|-------------|
| **누가 관리** | 사용자가 직접 | 사용자가 직접 | Claude가 자동 |
| **성격** | 프로젝트 지침서 | 글로벌 지침서 | 대화에서 학습한 기억 |
| **적용 범위** | 해당 프로젝트만 | 모든 프로젝트 공통 | 프로젝트별 분리 |
| **저장 위치** | 프로젝트 루트 | `~/.claude/CLAUDE.md` | `~/.claude/projects/<프로젝트>/memory/` |
| **편집 방법** | 직접 편집 또는 `/init` | `/memory` 명령어 | "기억해줘" / "잊어줘" |
| **로드** | 매 세션 전체 | 매 세션 전체 | 매 세션 MEMORY.md 첫 200줄 |

**Auto Memory 사용법:**
- **"기억해줘"라고 말하면** Claude가 해당 내용을 메모리에 저장한다. 별도의 복잡한 절차 없이 자연어로 요청하면 된다.
- **"잊어줘"라고 말하면** Claude가 해당 메모리를 삭제한다.
- 사용자가 Claude를 교정하거나 반복되는 패턴을 보이면 Claude가 알아서 기억하기도 한다.
- 저장된 메모리 파일은 `~/.claude/projects/<프로젝트>/memory/` 에서 직접 확인할 수 있다.
- 비활성화: 설정에서 `"autoMemoryEnabled": false`

**`/memory` 명령어:**
- `/memory`는 Auto Memory가 아니라 **`~/.claude/CLAUDE.md`** (글로벌 지침서)를 편집하는 명령어이다.
- 이 파일에 적은 내용은 **모든 프로젝트에서 공통으로 적용**된다.
- 예: 항상 한국어로 답해줘, 커밋 메시지는 한글로 작성해줘 등
