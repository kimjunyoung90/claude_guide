# Claude Code 실전 가이드 — 보충 문서

---

## 1. 프로젝트 루트에서 실행

Claude Code는 실행된 디렉토리를 기준으로 파일을 탐색한다.

- 일반적으로 **프로젝트 루트 디렉토리**에서 실행하는 것이 자연스럽다.
- 다른 디렉토리를 추가로 탐색하려면 `--add-dir` 옵션을 사용할 수 있다.

---

## 2. `/init`으로 CLAUDE.md 생성

`/init` 명령어를 실행하면 CLAUDE.md 파일이 생성된다.

### CLAUDE.md란?

- Claude가 실행될 때 **무조건 읽는 지침서**이다.
- 매번 모든 파일을 읽지 않아도 프로젝트 전체를 빠르게 파악할 수 있게 해준다.
- 범용 규칙, 프로젝트 구조, 해야 할 일, 하지 말아야 할 일 등 **범용적인 내용**을 담는다.

### 주의사항

- 항상 로드되는 파일이므로 **내용이 많을수록 토큰 소비가 커진다.**
- 세부 지침은 `.claude/rules/` 디렉토리에 주제별로 분리하여 관리하는 것을 권장한다.
- `paths`가 없는 rule은 CLAUDE.md처럼 세션 시작 시 **항상 로드**된다. `paths`가 있는 rule은 매칭된 경로의 파일을 **읽을 때만 로드**되어 토큰을 절약할 수 있다.
- 따라서 `paths` 없는 rule을 많이 만들면 CLAUDE.md에 다 넣는 것과 토큰 소비 차이가 없으므로, `paths` 스코핑을 잘 활용하는 것이 중요하다.
- 하위 디렉토리에 CLAUDE.md를 두는 방식도 지원되지만, 공식적으로는 `.claude/rules/` 사용을 더 권장한다.

### `.claude/rules/` 예시

```
프로젝트/
├── .claude/
│   └── rules/
│       ├── code-style.md        ← 항상 로드
│       ├── testing.md           ← 항상 로드
│       └── frontend/
│           └── react.md         ← src/components/ 작업 시에만 로드
```

경로 스코핑이 적용된 규칙 파일 예시:

```markdown
---
paths:
  - "src/components/**/*.tsx"
---

# React 컴포넌트 규칙
- 컴포넌트는 함수형으로 작성한다.
- Props 타입을 반드시 명시한다.
```

## 3. 핵심 단축키

### Shift + Tab — 모드 전환

Claude Code에는 세 가지 권한 모드가 있다.

| 모드 | 설명 |
|------|------|
| **Ask permissions (기본)** | 도구 사용 시 사용자에게 승인을 요청한다. |
| **Auto accept edits** | 파일 수정(Edit, Write)만 자동 승인. Bash 명령어 등 나머지 도구는 여전히 승인을 요청한다. |
| **Plan 모드** | 파일 작성/수정 권한 없음. 계획 수립과 검토만 진행한다. |

`Shift + Tab`으로 세 모드를 순환 전환할 수 있다.

#### `--dangerously-skip-permissions`

위 세 가지 모드 외에, CLI 플래그로만 활성화할 수 있는 **Bypass permissions** 모드가 있다.

```bash
claude --dangerously-skip-permissions
```

- Bash 명령어, 파일 수정, 웹 요청 등 거의 모든 도구를 **승인 없이 실행**한다.
- 단, `.git/`, `.claude/`, `.vscode/`, `.idea/` 등 보호 디렉토리에 대한 쓰기는 여전히 승인이 필요하다.
- **컨테이너나 VM 같은 격리된 환경에서만 사용**할 것을 공식 문서에서 경고하고 있다.
- Shift + Tab으로는 접근할 수 없으며, 실수로 토글되는 것을 방지하기 위해 의도적으로 분리되어 있다.

#### Plan 모드가 중요한 이유

1. **비용 절감** — 바로 코드 생성 모드로 들어가면, 의도와 다르게 작성된 경우 원복해야 하고 그만큼 비용이 낭비된다. 이 작업을 2~3번만 반복하게 되면 비용이 크게 불어난다.
2. **컨텍스트 보존** — 대화가 길어질수록 이전 컨텍스트가 압축되거나 삭제된다. 계획을 먼저 세우면 실제 코딩 단계에서 컨텍스트 윈도우를 코드 생성에 온전히 사용할 수 있다.
3. **실행 정확도 향상** — 한 번에 모든 걸 해결하려다 중간에 길을 잃는 것을 방지한다.

### Escape — 작업 중지 / 되돌리기

| 동작 | 설명 |
|------|------|
| **1번 누르기** | 현재 진행 중인 작업을 중지한다. |
| **2번 누르기** | **rewind/summarize** 다이얼로그가 열린다. 코드와 대화를 이전 시점으로 복원하거나, 특정 메시지부터 요약할 수 있다. |

### 기타 단축키

| 단축키 | 설명 |
|--------|------|
| 스크린샷 붙여넣기 | 터미널에 스크린샷을 직접 추가할 수 있다. |
| `!` + 명령어 | Claude Code 안에서 bash 명령어를 실행할 수 있다. (예: `!git status`) |

## 4. 필수 명령어

### 컨텍스트 관리

| 명령어 | 설명 |
|--------|------|
| `/clear` | 컨텍스트를 초기화한다. 쌓인 대화를 리셋할 때 사용. |
| `/context` | 현재 토큰 사용량을 확인한다. |
| `/compact` | 컨텍스트를 압축한다. 대화가 길어졌을 때 유용. |

### 세션 관리

| 명령어 | 설명 |
|--------|------|
| `/resume` | 이전 세션으로 복귀한다. `claude --resume`으로도 동일하게 사용 가능. |
| `/export` | 현재 대화를 텍스트 파일로 내보내거나 클립보드에 복사한다. |

### 설정

| 명령어 | 설명 |
|--------|------|
| `/model` | 사용할 모델을 선택한다. |
| `/mcp` | 사용 중인 MCP 서버 목록을 확인한다. |

---

## 5. MCP 사용 주의사항

MCP는 활성화되어 있는 것만으로도 컨텍스트를 차지한다.

- 현재는 **MCP 사용을 지양**하는 것을 추천한다.
- 꼭 필요한 경우, 필요한 기능만 담은 MCP를 직접 만들어 사용한다.
- 또는 MCP 대신 **스킬을 만들어** 동일한 기능을 구현하는 방식을 권장한다.

---

## 6. 반복 작업 자동화

반복적인 작업은 **Custom Command(스킬)**로 만들어 사용한다.

매번 같은 지시를 반복하지 않아도 슬래시 명령어 하나로 실행할 수 있다.
