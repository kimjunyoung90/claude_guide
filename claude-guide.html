CLAUDE.md — Claude가 자동으로 읽는 기본 지침서

직접 작성하거나 /init 으로 생성. 상세 지침은 rules로 분리하고, CLAUDE.md에는 핵심 + 참조만

wehago/
├── ...
├── .claude/
│   └── rules/
│       ├── code-style.md
│       ├── testing.md
│       └── frontend.md
└── CLAUDE.md
# 프로젝트 개요
Spring Boot 기반 위하고 플랫폼. Java 17, Gradle 빌드.

# 빌드 & 테스트
- `./gradlew build` — 전체 빌드
- `./gradlew test` — 전체 테스트

# 핵심 규칙
- DTO 네이밍: {동사}{도메인}Request/Response
- System.out.println 사용 금지 → Logger 사용
- 한국어로 응답

# 세부 규칙
- 코드 스타일 @.claude/rules/code-style.md
- 테스트 작성 @.claude/rules/testing.md
- 프론트엔드 @.claude/rules/frontend.md
한번 적어두면, 매번 말 안 해도 알아서 따른다.

Skills — "자주 하는 일을 명령어로"

SKILL.md는 500줄을 넘기지 말라고 권고, 넘으면 별도 파일로 분리

Skill 폴더 구조

.claude/skills/
└── weekly-report/
    ├── SKILL.md ← 메인 지침 (필수, 500줄 이하)
    ├── reference.md ← 상세 내용 (넘으면 분리)
    ├── examples/ ← 예시 (선택)
    └── scripts/ ← 실행 스크립트 (선택)
        └── helper.py

SKILL.md 예시

---
name: weekly-report
description: 이번 주 업무 보고서를 작성합니다. 주간 보고, 업무 정리, 이번 주 한 일 요약이 필요할 때 사용하세요.
---

금주 처리 티켓 확인
주간 보고서 초안 생성

호출 방식

직접 호출

› /weekly-report

자동 호출 (description 매칭)

› 이번 주 보고서 만들어줘.

● Skill(weekly-report)
  └ Successfully loaded skill

핵심: Claude는 description 필드를 보고 자동 호출 여부를 판단합니다.

무료 Skill 목록: https://skills.sh/hot

매번 같은 지시를 반복하지 않아도 된다.

Subagents — 별도 컨텍스트에서 일하는 독립 작업자

왜 Subagent가 필요한가?

직접 실행 시

› Confluence에서 결제 서비스 정책 찾아줘.

📄 문서 1 탐색... ← 컨텍스트에 쌓임
📄 문서 2 탐색... ← 컨텍스트에 쌓임
📄 문서 3 탐색... ← 컨텍스트에 쌓임
⚠ 탐색 과정 전부가 컨텍스트를 차지

Subagent에게 위임 시

› @search-agent 결제 서비스 정책 찾아줘.

→ (별도 컨텍스트에서 탐색 중...)
✓ 결과 요약만 내 컨텍스트에 반환

만드는 법

.claude/agents/
└── search-agent.md
---
name: search-agent
description: Confluence에서 문서를 검색합니다
tools: Read, Grep, Glob
model: haiku
---

Confluence에서 문서를 검색하고
핵심 내용만 요약해서 보고합니다.

호출 방식

@멘션: @search-agent 결제 서비스 정책 찾아줘
자연어: search-agent 써서 찾아줘
자  동: Claude가 description 보고 알아서 위임

모델 선택: 단순 검색 → Haiku | 코드 리뷰 → Sonnet | 복잡한 설계 → Opus

병렬 실행: 독립 작업은 동시에 실행 가능
순차 6분 → 병렬 2분

탐색 과정은 Subagent에게, 결과만 내 컨텍스트에.

› JIRA-1234 버그를 에이전트 팀을 구성해서 수정해줘.

  - 분석 담당: Jira 티켓 확인, 원인 파악
  - 프론트 담당: 화면 쪽 원인 탐색 및 수정
  - 백엔드 담당: API 쪽 원인 탐색 및 수정
  - 테스트 담당: 수정사항 기반 테스트 작성

  서로 발견한 내용을 공유하면서 진행해.
