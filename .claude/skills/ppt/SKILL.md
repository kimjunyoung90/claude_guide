# PPT 생성 스킬

`/frontend-slides` 스킬을 사용하여 프레젠테이션을 생성한다.

## 작업 흐름

1. 사용자의 요청을 파악한다.
2. `/frontend-slides` 스킬을 호출하여 HTML 프레젠테이션을 생성한다.
3. 아래 작업 원칙을 반드시 준수한다.

---

## 작업 원칙

### 반드시 지킬 것
- **내용 수정 전 반드시 방향 검증** — 사용자와 내용 및 방향을 먼저 확인한다.
- **관련 문서는 병렬적으로 수정** — 하나만 수정하고 나머지를 빠뜨리지 말 것
- **공식 문서 기반** — 내용의 정확성은 공식 문서로 검증할 것

### 하지 말 것
- 검증 없이 임의로 내용 추가/수정하지 않기
- 불확실한 정보를 넣지 않기 (확인 안 된 내용은 반드시 공식문서 확인)

---

## 디자인 원칙

- 컨텐츠는 **크게크게** — 잘 보이도록
- 시각화는 **단순 명확하게** — 숫자 비교, 색상 게이지 등 직관적 표현
- 터미널 UI를 HTML로 재현 (깜빡이는 커서, 연결 상태 등)
- 인터랙션 요소: 클릭 가능한 모달
- 폴더 구조는 같은 슬라이드 안에서 보여주기
- 공식문서 링크: 우하단, 빨간색, z-index:10으로 클릭 가능하게

### 시각화 원칙
- **글자 없이도 의미가 전달**되어야 한다 — 텍스트 나열이 아니라 시각 자료가 핵심
- **이모지를 시각 요소로 쓰지 않는다** — 전문성이 떨어져 보인다
- **전체 슬라이드 영역을 활용**한다 — 한쪽에 치우치지 않게
- 차트/그래프는 **색상 대비로 핵심 구간을 강조**한다 (예: 빨간색 vs 회색)
- Before/After 비교가 항상 필요한 건 아니다 — 하나의 시각 자료로 충분하면 그것만 사용

### 넣지 말 것
- 과금/비용 관련 정보 — 핵심 메시지가 흐려진다
- 억지스러운 비유 표현 — 와닿지 않으면 오히려 역효과
- 선형적인 히트맵(높이가 같은 막대의 색상만 다른) — 바 차트 높이로 차이를 표현할 것

## 스크립트 작성 원칙

PPT와 함께 발표 스크립트를 작성할 때:

- 대본처럼 읽을 수 있는 문장으로 작성
- 발화 시간 표기
- Q&A 대비 섹션 포함
- 비전문가도 이해할 수 있는 어투

---

## 문서 관리 원칙

이 프로젝트의 문서 구조와 역할:

| 문서 | 역할 |
|------|------|
| `slides.md` | 슬라이드 내용 (키워드, 표, 구조) |
| `script.md` | 발표 스크립트 (말할 내용 전체) |
| `supplement.md` | PPT에 추가할 내용의 **대기열**. 반영 후 제거 |
| `claude-guide.html` | 실제 프레젠테이션 HTML |

- **보충문서(supplement.md)에서 제거할 때는 반드시 script.md에 해당 내용이 반영되어 있어야 한다**
- slides.md, script.md, claude-guide.html은 항상 동기화한다
- 제목이나 시간 변경 시 3개 파일 모두 반영한다
