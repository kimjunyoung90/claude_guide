## Slide 1: Claude란?

**모델 종류**

| 모델 | 특징 | 적합한 용도 |
|---|---|---|
| **Opus** | 가장 강력, 깊은 사고력 | 복잡한 분석, 어려운 코딩 |
| **Sonnet** | 성능과 속도의 균형 | 일반 코딩, 일상 업무 |
| **Haiku** | 가장 빠르고 저렴 | 단순 작업, 빠른 응답 |

**할 수 있는 것들**

- 코드 작성 / 분석 / 리팩토링
- 문서 작성 / 요약 / 번역
- 데이터 분석 / 정리
- 테스트 케이스 생성
- 마크업 생성 / 수정
- 기획서 검토 / 피드백

> 코딩만이 아니라, 모든 업무에 활용 가능

## Slide 3: 알아두면 좋은 기본 개념

**토큰 (Token)**
- AI가 읽고 쓰는 단위
- 영어는 짧은 단어 하나가 대략 1토큰
- 한글은 같은 의미라도 더 많은 토큰 소모

**컨텍스트 (Context)**
- AI가 답변 생성 시 참고할 수 있는 텍스트의 양
- 대화 내용 + 파일 내용 + 설정 = 전부 컨텍스트에 들어감
- 컨텍스트가 가득 차면 → **대화 내용을 잊어버리거나, 중요한 맥락을 못 짚는 경우 발생**

> 불필요한 정보가 컨텍스트를 차지할수록, AI 성능이 떨어진다.

---

## Slide 4: 왜 불필요한 정보가 성능을 떨어뜨리는가?

1. 컨텍스트에 정보가 많으면 AI가 **뭐가 중요한지 판단하기 어려워짐**
2. 필요한 정보가 불필요한 정보에 **묻혀서 놓침**
3. 사람도 마찬가지 — 보고서 100페이지에서 핵심 3줄 찾기 vs 핵심만 정리된 1페이지

> 많이 넣는다고 좋은 게 아니라, 핵심만 넣어야 AI가 제대로 동작한다.

## Slide 5: Claude Code의 핵심 기능들

| 기능 | 한마디 정의 |
|---|---|
| **CLAUDE.md** | Claude가 자동으로 읽는 기본 지침서 |
| **Skills** | 자주 쓰는 작업을 명령어로 만들어두는 것 (`/deploy`, `/review`) |
| **MCP** | 외부 서비스(Jira, Confluence, DB 등)를 Claude에 연결하는 통로 |
| **Subagents** | 큰 작업을 맡길 수 있는 부하 직원 (결과만 보고받음) |
| **Agent Teams** | 여러 Claude가 팀을 이뤄 동시에 일하는 것 |
| **Hooks** | 특정 상황에 자동으로 실행되는 스크립트 |
| **Plugins** | 위 기능들을 묶어서 팀끼리 공유하는 패키지 |

**폴더 구조:**
```
업무자동화/
├── .claude/
│   ├── settings.json      ← 클로드 설정
│   ├── hooks/             ← Hook 실행 시 사용될 스크립트(코드)
│   │   └── auto-format.sh
│   ├── skills/            ← Skills
│   │   └── review/
│   │       └── SKILL.md
│   ├── agents/            ← Subagents
│   │   └── security-auditor.md
│   └── rules/             ← 추가 규칙
│       └── testing.md
├── .mcp.json              ← MCP 서버 설정
└── CLAUDE.md              ← 기본 지침서
```

## Slide 6: CLAUDE.md — Claude가 자동으로 읽는 기본 지침서

간단한 기본 내용만 들어있고, **200줄을 넘기지 말라고 권고**

**개발팀 예시:**
```
wehago/
├── src/
├── build.gradle
└── CLAUDE.md     ← 여기에 위치
```
```
# CLAUDE.md
- ./gradlew test — 전체 테스트
- DTO: {동사}{도메인}Request 형식
- System.out.println 금지
```

**QA팀 예시:**
```
qa-workspace/
├── 테스트케이스/
├── 버그리포트/
└── CLAUDE.md     ← 여기에 위치
```
```
# CLAUDE.md
- TC 작성 시 정상/비정상/경계값 반드시 포함
- 버그 리포트는 재현 절차 필수
- 우선순위: P1(핵심) / P2(주요) / P3(엣지)
```

> 한번 적어두면, 매번 말 안 해도 Claude가 알아서 따른다.

## Slide 7: Skills — "자주 하는 일을 명령어로"

반복되는 워크플로우를 `/명령어`로 등록

**개발자용:**
- `/code-review` → git diff 기반, 🔴Critical / 🟠Warning / 🟡Suggestion 분류
- `/api-docs` → Controller 분석 → API 명세서 자동 생성
- `/commit` → Conventional Commits 형식 커밋 메시지 생성

**QA용:**
- `/tc-generator` → 기능 명세서에서 테스트 케이스 자동 도출
  - 정상/비정상/경계값/예외 케이스 분류
  - 우선순위: P1(핵심) / P2(주요) / P3(엣지)
  - 출력: TC-ID | 우선순위 | 사전조건 | 절차 | 기대결과

**일반 업무용:**
- `/weekly-report` → 이번 주 git 커밋/PR 분석 → 주간 보고서 초안 생성
- `/meeting-notes` → 회의록 정리 → 액션 아이템 추출

> "매번 같은 지시를 반복하지 않아도 된다"

## Slide 8: MCP — "외부 서비스 연결 통로"

Claude Code가 **외부 도구에 직접 접근**할 수 있게 해주는 연결 장치

- **Jira** → 티켓 조회 / 상태 업데이트
- **Confluence** → 페이지 읽기 / 보고서 자동 작성
- **사내 메신저** → 메시지 전송 / 채널 읽기 (MCP 커스텀 연동)
- **DB** → 쿼리 실행 / 데이터 조회
- **GitHub** → PR 생성 / 이슈 관리

**실제 프롬프트 예시:**
- "orders 테이블에서 최근 7일간 일별 주문 건수 조회해줘"
- "이번 스프린트의 Jira 티켓 현황을 Confluence 주간보고 페이지에 정리해줘"
- "사내 메신저 #dev-alerts 채널에서 오늘 에러 관련 메시지 찾아줘"
- "이 이슈에 대한 PR을 만들어줘"

**주의: MCP는 대화 시작 시 활성화된 모든 MCP의 사용법을 컨텍스트에 포함시킴**
→ 안 쓰는 MCP가 켜져 있으면 컨텍스트를 잡아먹는 주범이 됨
→ `/mcp` 명령어로 토큰 사용량 확인 가능, 미사용 MCP는 반드시 비활성화

> 브라우저 열지 않아도, Claude가 알아서 연동한다.

## Slide 9: Subagents & Agent Teams

**Subagents** — 부하 직원에게 일 시키기

| 에이전트 | 역할 | 예시 |
|---|---|---|
| security-auditor | 보안 감사 | SQL 인젝션, XSS, 하드코딩 시크릿 탐지 |
| perf-analyzer | 성능 분석 | N+1 쿼리, 캐싱 기회, 불필요한 반복 식별 |
| qa-analyzer | QA 분석 | 변경 코드 기반 영향 범위 분석, 리그레션 포인트 도출 |

**Agent Teams** — 여러 Claude가 동시에 협업

```
결제 모듈을 리팩토링할 에이전트 팀을 만들어줘.
- API 담당: 결제 API 엔드포인트와 DTO 리팩토링
- DB 담당: 결제 테이블 마이그레이션 스크립트 작성
- 테스트 담당: 결제 관련 통합 테스트 작성
서로 발견한 내용을 공유하면서 진행해.
```

> 혼자 다 안 해도 된다. 팀을 꾸려서 시키면 된다.

## Slide 10: Hooks & Plugins

**Hooks** — 이벤트가 발생하면 자동 실행

| 트리거 | 동작 | 효과 |
|---|---|---|
| 파일 수정 후 | `prettier --write` 자동 실행 | 포맷팅 신경 안 써도 됨 |
| .env 파일 수정 시도 | **수정 차단** + 경고 메시지 | 보호 파일 실수 방지 |
| .java 파일 수정 후 | `gradlew compileJava` 자동 실행 | 컴파일 에러 즉시 감지 |

**Plugins** — 위 기능들을 하나로 묶어 배포
- Skills + Hooks + MCP를 한 묶음으로 패키징
- Git repo로 공유 → 팀 전체가 같은 환경
- `claude --plugin-dir ./my-plugin` 또는 `/plugin install`로 설치

> 반복 작업은 사람이 아니라 시스템이 한다.

## Slide 11: 컨텍스트 관리 원칙

1. **CLAUDE.md는 200줄 이하** → 핵심 규칙만 (빌드 명령, 코딩 컨벤션)
2. **SKILL.md는 500줄 이하** → 필요할 때만 로드되게
3. **불필요한 정보는 삭제** → "혹시 몰라서" 넣은 내용이 성능을 깎는다
4. **스킬로 분리** → 모든 정보를 한 파일에 넣지 말고, 작업별로 나누기
5. **미사용 MCP는 비활성화** → 안 쓰는 MCP도 컨텍스트를 차지함

> 짧고 핵심적으로 쓸수록, AI가 더 정확하게 동작한다.
