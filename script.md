# Claude 교육 — 발표 스크립트

총 예상 시간: 약 25~30분

---

## Slide 1: Claude란? (약 2분)

안녕하세요. 오늘은 Claude에 대해 이야기해보겠습니다.

Claude는 Anthropic이라는 회사에서 만든 AI입니다. 모델이 여러 종류가 있는데요. Opus는 가장 강력하고 깊은 사고가 가능한 모델이고, Sonnet은 성능과 속도의 균형이 잡힌 모델, Haiku는 가장 빠르고 저렴한 모델입니다. 용도에 따라 선택해서 쓸 수 있습니다.

그리고 Claude가 할 수 있는 건 코딩만이 아닙니다. 코드 작성이나 분석은 물론이고, 문서 작성, 번역, 데이터 분석, 테스트 케이스 생성, 마크업 수정, 기획서 피드백까지 — 사실상 모든 업무에 활용이 가능합니다.

---

## Slide 2: Claude, 어떻게 쓸 수 있나? (약 1분 30초)

Claude를 쓸 수 있는 방법은 크게 세 가지가 있습니다.

첫 번째는 웹 Claude입니다. 브라우저에서 대화하듯이 질문하고 답변 받는 방식이고요. 두 번째는 Claude Desktop이라는 데스크톱 앱인데, 파일을 연동해서 쓸 수 있습니다. 이런 것들도 있다 정도로 알아두시면 됩니다.

오늘 집중적으로 볼 건 세 번째, Claude Code입니다. 이건 터미널에서 동작하는 AI 개발 도구인데, 여러분의 프로젝트에서 바로 동작하면서 파일을 직접 읽고 수정하고 생성까지 합니다.

---

## Slide 3: 알아두면 좋은 기본 개념 (약 2분 30초)

Claude Code를 잘 쓰려면 두 가지 개념을 알아야 합니다. 토큰과 컨텍스트입니다.

먼저 토큰입니다. AI가 텍스트를 읽고 쓸 때 사용하는 단위인데요. 영어는 짧은 단어 하나가 대략 1토큰입니다. 한글은 같은 의미라도 더 많은 토큰을 소모합니다. 왜 토큰이 중요하냐면, 이게 컨텍스트와 직결되기 때문입니다.

컨텍스트는 AI가 답변 생성 시 참고할 수 있는 텍스트의 양입니다. 여러분이 AI에게 보내는 대화 내용, 파일 내용, 설정 — 이 모든 게 토큰으로 변환되어 컨텍스트 안에 들어갑니다.

문제는 이 컨텍스트가 가득 차면 생기는 겁니다. AI가 대화 내용을 잊어버리거나, 중요한 맥락을 못 짚는 경우가 발생합니다. 그리고 불필요한 정보가 컨텍스트를 차지할수록, AI 성능이 떨어집니다.

---

## Slide 4: 왜 불필요한 정보가 성능을 떨어뜨리는가? (약 1분 30초)

그러면 왜 불필요한 정보가 있으면 AI 성능이 떨어질까요? 이유는 간단합니다.

첫째, 컨텍스트에 정보가 많아지면 AI가 뭐가 중요한지 판단하기 어려워집니다. 전부 다 참고하려다 보니 집중이 안 되는 겁니다.

둘째, 정작 필요한 정보가 불필요한 정보에 묻혀서 놓치게 됩니다. 중요한 규칙이 있어도 주변에 쓸데없는 내용이 가득하면 그걸 제대로 캐치하지 못합니다.

사람도 마찬가지입니다. 보고서 100페이지에서 핵심 3줄을 찾는 것과, 핵심만 정리된 1페이지를 받는 것. 어느 쪽이 더 빠르고 정확할까요? AI도 똑같습니다.

많이 넣는다고 좋은 게 아니라, 핵심만 넣어야 AI가 제대로 동작합니다.

---

## Slide 5: 연구가 증명한 컨텍스트 관리의 중요성 (약 2분)

이건 제 개인 의견이 아니라, 실제 논문으로 증명된 내용입니다. "Repository Level Context Files Helpful for Coding Agent?"라는 연구인데요.

결과를 보시면, 컨텍스트 파일이 아예 없을 때를 기준으로 — AI가 자동으로 생성한 컨텍스트 파일을 넣었더니 오히려 작업 성공률이 2% 떨어졌습니다. 비용은 20% 이상 올라갔고요. 중복된 정보, 불필요한 정보가 AI를 방해한 겁니다.

반면에, 사람이 직접 작성한 간결한 컨텍스트 파일을 넣었을 때는 성공률이 19% 올라갔습니다. 필수 요구사항만 담았기 때문입니다.

많이 쓰는 게 중요한 게 아니라, 잘 쓰는 게 중요합니다.

---

## Slide 6: 실무에서의 컨텍스트 관리 원칙 (약 1분 30초)

그래서 실무에서 지켜야 할 원칙 네 가지를 정리했습니다.

첫째, CLAUDE.md는 200줄 이하로 유지합니다. 빌드 명령, 코딩 컨벤션 같은 핵심 규칙만 넣으세요.

둘째, SKILL.md는 500줄 이하로 유지합니다. 필요할 때만 로드되도록요.

셋째, 불필요한 정보는 삭제합니다. "혹시 몰라서" 넣은 내용이 성능을 깎습니다.

넷째, 스킬로 분리합니다. 모든 정보를 CLAUDE.md 하나에 다 넣지 말고, 작업별로 나눠서 관리하세요.

다섯째, 미사용 MCP는 비활성화하세요. 이건 뒤에서 자세히 설명드리겠지만, 안 쓰는 MCP도 켜져 있으면 컨텍스트를 차지합니다.

짧고 핵심적으로 쓸수록, AI가 더 정확하게 동작합니다.

## Slide 7: Claude Code의 핵심 기능들 (약 2분)

이제 Claude Code가 제공하는 핵심 기능들을 하나씩 살펴보겠습니다.

CLAUDE.md는 Claude가 자동으로 읽는 기본 지침서입니다. Skills는 자주 쓰는 작업을 슬래시 명령어로 만들어두는 겁니다. MCP는 Jira, Confluence, DB 같은 외부 서비스를 Claude에 연결하는 통로입니다. Subagents는 큰 작업을 맡길 수 있는 부하 직원이라고 생각하시면 됩니다. 결과만 보고받죠. Agent Teams는 여러 Claude가 팀을 이뤄서 동시에 일하는 겁니다. Hooks는 특정 이벤트가 발생하면 자동으로 실행되는 스크립트이고, Plugins는 이 모든 기능들을 하나로 묶어서 팀끼리 공유하는 패키지입니다.

오른쪽에 폴더 구조를 보시면, 이 기능들이 실제로 어디에 위치하는지 알 수 있습니다. .claude 폴더 안에 settings.json이 클로드 설정 파일이고, hooks 폴더에 실행될 스크립트, skills 폴더에 스킬 파일, agents 폴더에 에이전트 정의, rules 폴더에 추가 규칙이 들어갑니다. MCP 설정은 루트에 .mcp.json으로, CLAUDE.md도 루트에 위치합니다.

각각에 대해 좀 더 자세히 보겠습니다.

## Slide 6: CLAUDE.md — Claude가 자동으로 읽는 기본 지침서 (약 2분)

먼저 CLAUDE.md입니다. 이건 Claude가 자동으로 읽는 기본 지침서입니다. 작업 폴더에 이 파일을 두면, Claude가 매 대화를 시작할 때 자동으로 읽습니다. 간단한 기본 내용만 넣고, 200줄을 넘기지 말라고 권고하고 있습니다.

개발팀 예시를 보시면, wehago 폴더 안에 CLAUDE.md를 두고, 빌드 명령어나 DTO 네이밍 규칙, 금지 사항 같은 핵심만 적어둡니다.

QA팀도 마찬가지입니다. qa-workspace 폴더에 CLAUDE.md를 두고, TC 작성 규칙이나 버그 리포트 형식, 우선순위 기준을 적어두면 됩니다. 개발자만 쓸 수 있는 게 아닙니다.

한번 적어두면, 매번 말 안 해도 Claude가 알아서 따릅니다.

## Slide 9: Skills — "자주 하는 일을 명령어로" (약 2분 30초)

다음은 Skills입니다. 반복되는 워크플로우를 슬래시 명령어로 등록해두면, 한번에 실행할 수 있습니다.

개발자용으로는 /code-review를 치면 git diff를 기반으로 Critical, Warning, Suggestion을 분류해서 리뷰해줍니다. /api-docs는 Controller를 분석해서 API 명세서를 자동으로 만들어주고, /commit은 Conventional Commits 형식으로 커밋 메시지를 생성합니다.

QA분들도 주목해주세요. /tc-generator는 기능 명세서를 넣으면 테스트 케이스를 자동으로 도출해줍니다. 정상, 비정상, 경계값, 예외 케이스를 분류하고, 우선순위도 매겨서 TC-ID, 사전조건, 절차, 기대결과까지 표로 만들어줍니다.

일반 업무용으로는 /weekly-report로 이번 주 git 활동을 분석해서 주간 보고서 초안을 만들 수도 있고, /meeting-notes로 회의록을 정리하고 액션 아이템을 추출할 수도 있습니다.

참고로 슬래시 명령어로 직접 호출할 때, SKILL.md에 name 필드가 있으면 그 name으로 호출되고, name이 없으면 폴더명으로 호출됩니다.

매번 같은 지시를 반복하지 않아도 됩니다.

## Slide 8: MCP — "외부 서비스 연결 통로" (약 2분)

MCP는 Claude Code가 외부 도구에 직접 접근할 수 있게 해주는 연결 장치입니다.

Jira를 연결하면 티켓을 조회하거나 상태를 업데이트할 수 있고, Confluence를 연결하면 페이지를 읽거나 보고서를 자동으로 작성할 수 있습니다. 사내 메신저도 커스텀 MCP로 연동할 수 있고, DB에 직접 쿼리를 날리거나 GitHub PR을 만드는 것도 가능합니다.

실제로 이런 식으로 쓸 수 있습니다. "orders 테이블에서 최근 7일간 일별 주문 건수 조회해줘." 또는 "이번 스프린트의 Jira 티켓 현황을 Confluence 주간보고 페이지에 정리해줘." 이렇게 자연어로 말하면 Claude가 알아서 해당 서비스에 접근해서 처리합니다.

한 가지 주의사항이 있습니다. MCP는 대화를 시작하는 순간, 활성화된 모든 MCP의 사용법 정보를 컨텍스트에 포함시킵니다. 그래서 안 쓰는 MCP가 켜져 있으면 컨텍스트를 잡아먹는 주범이 됩니다. /mcp 명령어로 토큰 사용량을 확인할 수 있으니, 미사용 MCP는 반드시 비활성화하세요.

브라우저 열지 않아도, Claude가 알아서 연동합니다.

## Slide 11: Subagents & Agent Teams (약 2분)

Subagents는 전문가에게 일을 위임하는 개념입니다.

예를 들어 security-auditor 에이전트는 코드에서 SQL 인젝션, XSS, 하드코딩된 시크릿 같은 보안 취약점을 탐지합니다. perf-analyzer는 N+1 쿼리나 캐싱 기회를 분석하고, qa-analyzer는 변경된 코드의 영향 범위를 분석해서 리그레션 포인트를 도출합니다. 이 에이전트들은 각자 작업을 수행하고 결과만 보고해주기 때문에, 메인 대화의 컨텍스트를 오염시키지 않습니다.

Agent Teams는 한 단계 더 나아갑니다. 여러 Claude가 팀을 이뤄서 동시에 작업합니다. 예를 들어 결제 모듈을 리팩토링할 때, API 담당, DB 담당, 테스트 담당 세 명의 에이전트가 동시에 작업하면서 서로 발견한 내용을 공유합니다.

혼자 다 안 해도 됩니다. 팀을 꾸려서 시키면 됩니다.

## Slide 12: Hooks & Plugins (약 1분 30초)

Hooks는 특정 이벤트가 발생하면 자동으로 실행되는 스크립트입니다.

파일을 수정하면 prettier가 자동으로 돌아가서 포맷팅을 맞춰주고, .env 같은 보호 파일을 수정하려고 하면 아예 차단하고 경고 메시지를 띄웁니다. Java 파일을 수정하면 자동으로 컴파일을 돌려서 에러를 즉시 감지할 수도 있습니다.

Plugins는 이런 Skills, Hooks, MCP 설정을 하나로 묶어서 팀 전체에 배포하는 패키지입니다. Git repo에 올려두면 팀 전체가 같은 환경에서 작업할 수 있습니다.

반복 작업은 사람이 아니라 시스템이 합니다.

## Slide 13: 데모 — 지라 티켓 하나로 PR까지 (약 8~10분)

자, 지금까지 설명드린 내용이 실제로 어떻게 동작하는지 보여드리겠습니다.

제가 실무에서 구축한 워크플로우인데요. 지라 티켓 하나를 전달하면, Claude가 관련 프로젝트를 탐색하고, 코드를 분석하고, 수정한 다음, PR까지 생성합니다.

(데모 영상 재생)

보신 것처럼, 사람이 한 건 지라 티켓을 전달한 것뿐이고, 나머지는 Claude가 알아서 처리했습니다.

여러분의 업무에서도 이런 자동화가 가능합니다. 오늘 소개해드린 기능들을 활용해서, 각자의 업무에 맞는 워크플로우를 만들어보시기 바랍니다.

감사합니다.

## 예상 Q&A 대비

**Q: AI가 코드를 진짜 이해하는 건가요?**
> "이해한다기보다는, 엄청난 양의 코드를 학습해서 이 상황에서 가장 적절한 코드가 뭔지를 확률적으로 판단하는 겁니다. 근데 그 확률이 워낙 높아서, 사실상 이해하는 것처럼 보이는 거죠."

**Q: AI가 틀린 답을 주면 어떡하나요?**
> "당연히 틀릴 수 있습니다. 그래서 AI가 만든 결과물은 반드시 검증해야 합니다. 다만, 초안을 만들어주는 것만으로도 업무 시간이 크게 줄어듭니다."

**Q: ChatGPT랑 뭐가 다른가요?**
> "ChatGPT는 OpenAI에서 만든 모델이고, Claude는 Anthropic에서 만든 모델입니다. 각각 장단점이 있지만, 우리가 Claude를 쓰는 이유는 Claude Code라는 도구가 터미널에서 바로 프로젝트 작업이 가능하기 때문입니다."

**Q: 보안은 괜찮은가요?**
> "코드가 외부로 전송되는 것에 대한 우려가 있을 수 있습니다. Anthropic은 Zero Data Retention 옵션을 제공하고 있고, 사내 보안 정책에 맞춰서 운영하고 있습니다."
