# Claude 교육 — 발표 스크립트

총 예상 시간: 약 30분

## Slide 1: Claude란? (약 2분 30초)

안녕하세요. 오늘은 Claude에 대해 이야기해보겠습니다.

Claude는 Anthropic이라는 회사에서 만든 AI입니다. 모델이 여러 종류가 있는데요. Opus는 가장 강력하고 깊은 사고가 가능한 모델이고, Sonnet은 성능과 속도의 균형이 잡힌 모델, Haiku는 가장 빠르고 저렴한 모델입니다. 용도에 따라 선택해서 쓸 수 있습니다.

그리고 Claude가 할 수 있는 건 코딩만이 아닙니다. 코드 작성이나 분석은 물론이고, 문서 작성, 번역, 데이터 분석, 테스트 케이스 생성, 마크업 수정, 기획서 피드백까지 — 사실상 모든 업무에 활용이 가능합니다.

## Slide 2: Claude 제공 형태 (약 1분 30초)

Claude는 크게 세 가지 형태로 서비스를 제공합니다.

첫 번째는 웹 Claude입니다. 브라우저에서 대화하듯이 질문하고 답변 받는 방식이고요. 두 번째는 Claude Desktop이라는 데스크톱 앱인데, 파일을 연동해서 쓸 수 있습니다. 이런 것들도 있다 정도로 알아두시면 됩니다.

오늘 집중적으로 볼 건 세 번째, Claude Code입니다. 이건 터미널에서 동작하는 AI 개발 도구인데, 여러분의 프로젝트에서 바로 동작하면서 파일을 직접 읽고 수정하고 생성까지 합니다.

## Slide 3: 알아두면 좋은 기본 개념 (약 2분 30초)

Claude Code를 잘 쓰려면 두 가지 개념을 알아야 합니다. 토큰과 컨텍스트입니다.

먼저 토큰입니다. AI가 텍스트를 읽고 쓸 때 사용하는 단위인데요. 영어는 짧은 단어 하나가 대략 1토큰입니다. 한글은 같은 의미라도 더 많은 토큰을 소모합니다. 왜 토큰이 중요하냐면, 이게 컨텍스트와 직결되기 때문입니다.

컨텍스트는 AI가 답변 생성 시 참고할 수 있는 텍스트의 양입니다. 여러분이 AI에게 보내는 대화 내용, 파일 내용, 설정 — 이 모든 게 토큰으로 변환되어 컨텍스트 안에 들어갑니다.

문제는 이 컨텍스트가 가득 차면 생기는 겁니다. AI가 대화 내용을 잊어버리거나, 중요한 맥락을 못 짚는 경우가 발생합니다. 그리고 불필요한 정보가 컨텍스트를 차지할수록, AI 성능이 떨어집니다.

## Slide 4: 컨텍스트가 길어지면 무슨 일이 생기는가? (약 1분 30초)

그러면 왜 불필요한 정보가 있으면 AI 성능이 떨어질까요?

첫째, 정보가 많아지면 AI가 뭐가 중요한지 판단하기 어려워집니다.

둘째, 필요한 정보가 불필요한 정보에 묻혀서 놓치게 됩니다.

셋째, Lost in the Middle이라는 현상이 있습니다. 스탠포드 연구에서 밝혀진 건데, AI는 컨텍스트의 처음과 끝은 잘 기억하지만 중간에 있는 정보는 놓치는 경향이 있습니다.

사람도 마찬가지입니다. 보고서 100페이지에서 핵심 3줄 찾기와, 핵심만 정리된 1페이지. AI도 똑같습니다. 핵심만 넣어야 제대로 동작합니다.

## Slide 5: Claude Code의 핵심 기능들 (약 2분)

Claude Code를 사용할 때 한 가지 알아두실 점은, 프로젝트 루트 디렉토리에서 실행해야 한다는 겁니다. Claude Code는 실행된 디렉토리를 기준으로 파일을 탐색하기 때문에, 프로젝트 루트에서 시작해야 전체 구조를 제대로 파악할 수 있습니다.

이제 Claude Code가 제공하는 핵심 기능들을 하나씩 살펴보겠습니다.

CLAUDE.md는 Claude가 자동으로 읽는 기본 지침서입니다. Skills는 자주 쓰는 작업을 슬래시 명령어로 만들어두는 겁니다. MCP는 Jira, Confluence, DB 같은 외부 서비스를 Claude에 연결하는 통로입니다. Subagents는 별도 컨텍스트에서 일하는 독립 작업자라고 생각하시면 됩니다. 결과만 보고받죠. Agent Teams는 여러 Claude가 팀을 이뤄서 동시에 일하는 겁니다. Hooks는 특정 이벤트가 발생하면 자동으로 실행되는 스크립트이고, Plugins는 이 모든 기능들을 하나로 묶어서 공유하는 패키지입니다.

오른쪽에 폴더 구조를 보시면, 이 기능들이 실제로 어디에 위치하는지 알 수 있습니다. .claude 폴더 안에 settings.json이 클로드 설정 파일이고, hooks 폴더에 실행될 스크립트, skills 폴더에 스킬 파일, agents 폴더에 에이전트 정의, rules 폴더에 추가 규칙이 들어갑니다. MCP 설정은 루트에 .mcp.json으로, CLAUDE.md도 루트에 위치합니다.

각각에 대해 좀 더 자세히 보겠습니다.

## Slide 6: CLAUDE.md — Claude가 자동으로 읽는 기본 지침서 (약 2분)

먼저 CLAUDE.md입니다. 이건 Claude가 자동으로 읽는 기본 지침서입니다. `/init` 명령어를 실행하면 자동으로 생성됩니다. 작업 폴더에 이 파일을 두면, Claude가 매 대화를 시작할 때 자동으로 읽습니다. 간단한 기본 내용만 넣고, 200줄을 넘기지 말라고 권고하고 있습니다.

예시를 보시면, wehago 폴더 안에 CLAUDE.md를 두고, 빌드 명령어나 DTO 네이밍 규칙, 금지 사항 같은 핵심만 적어둡니다. 개발뿐 아니라 QA, 퍼블리싱 등 어떤 작업이든 동일하게 활용할 수 있습니다.

한번 적어두면, 매번 말 안 해도 Claude가 알아서 따릅니다.

여기서 중요한 점이 하나 있습니다. CLAUDE.md는 항상 로드되는 파일이기 때문에, 내용이 많을수록 토큰 소비가 커집니다. 모든 규칙을 CLAUDE.md 하나에 다 넣으면, 프론트엔드만 수정하는데 DB 규칙이나 배포 규칙까지 매번 읽게 됩니다. 그래서 상세 지침은 `.claude/rules/` 디렉토리에 분리하고, CLAUDE.md에는 핵심 규칙과 함께 `@` 문법으로 rules 파일을 참조하는 형태로 작성하는 것을 권장합니다. 예를 들어 `@.claude/rules/code-style.md`처럼 적으면 Claude가 해당 파일을 자동으로 읽습니다.

rules 파일에는 `paths`라는 설정을 줄 수 있는데, 이걸 지정하면 해당 경로의 파일을 작업할 때만 규칙이 로드됩니다. 예를 들어 프론트엔드 규칙은 `src/components/` 작업 시에만, DB 규칙은 `prisma/` 작업 시에만 로드되게 할 수 있습니다. `paths`가 없으면 CLAUDE.md처럼 항상 로드되니까, `paths` 스코핑을 잘 활용하는 게 토큰 절약의 핵심입니다.

## Slide 7: Plan 모드 (약 2분)

Claude Code에는 세 가지 작업 모드가 있습니다. Shift + Tab으로 전환할 수 있는데요. Ask permissions 모드는 기본 모드로, 파일 편집이든 명령 실행이든 모든 작업에 승인을 요청합니다. Auto accept edits 모드는 파일 편집만 자동으로 승인하고, Bash 명령어 같은 건 여전히 승인이 필요합니다. 그리고 Plan 모드는 코드를 작성하지 않고 계획만 세우는 모드입니다.

참고로 이 세 가지 외에 `--dangerously-skip-permissions`라는 옵션도 있는데, 이건 모든 승인을 건너뛰는 모드입니다. 컨테이너나 VM 같은 격리된 환경에서만 사용해야 하고, Shift + Tab으로는 접근할 수 없게 의도적으로 분리되어 있습니다.

이 중에서 제가 가장 강조하고 싶은 건 Plan 모드입니다. 이건 Claude Code를 잘 쓰기 위한 가장 중요한 습관입니다.

Plan 모드가 중요한 이유는 세 가지입니다.

첫째, 비용 절감입니다. 복잡한 작업을 바로 코드 작성 모드로 시키면 의도와 다르게 작성될 확률이 높습니다. 그러면 원복하고 다시 요청해야 하고, 그 과정에서 토큰이 이중으로 소비됩니다. 2~3번만 반복해도 비용이 3배 이상 불어납니다.

둘째, 컨텍스트 보존입니다. 대화가 길어질수록 이전 컨텍스트가 압축되거나 삭제됩니다. Plan 없이 바로 코딩하면 시행착오가 컨텍스트를 빠르게 소진시킵니다. 계획을 먼저 세우면 실제 코딩 단계에서 컨텍스트 윈도우를 온전히 활용할 수 있습니다.

셋째, 실행 정확도 향상입니다. 한 번에 모든 걸 해결하려다 중간에 길을 잃는 걸 방지합니다. Plan 모드에서 Claude가 계획을 제시하면 사용자가 검토하고, 방향이 틀리면 코드 작성 전에 바로잡을 수 있습니다.

실전에서는 이런 흐름으로 씁니다. Shift + Tab으로 Plan 모드에 진입하고, 작업을 요청합니다. Claude가 계획을 제시하면 리뷰하고 피드백을 줍니다. "JWT 대신 세션 기반으로 해줘", "미들웨어 구조를 바꿔줘" 이런 식으로요. 계획이 확정되면 Shift + Tab으로 코드 작성 모드로 전환하고 "위 계획대로 구현해줘"라고 하면 됩니다.

간단한 기준을 드리면, 한 줄 수정이나 오타 fix는 Plan 모드 없이 바로 해도 됩니다. 함수 추가나 리팩토링은 권장이고, 새 기능이나 아키텍처 변경은 Plan 모드가 필수입니다. 코드를 쓰기 전에 방향을 맞추면 재작업이 사라집니다.

## Slide 8: Skills — 구조와 파일 예시 (약 1분)

다음은 Skills입니다. 반복되는 작업을 슬래시 명령어로 등록해두면, 한번에 실행할 수 있습니다.

먼저 구조를 보겠습니다. `.claude/skills/` 폴더 안에 스킬 이름으로 디렉토리를 만들고, 그 안에 파일들을 작성합니다.

SKILL.md가 필수 파일입니다. 스킬의 개요와 Claude가 따를 핵심 지침을 담습니다. 그 외에 reference.md 같은 보조 문서를 둘 수 있는데, SKILL.md에 다 넣기엔 긴 내용을 분리해두는 용도입니다. 선택사항이고, 필요할 때만 Claude가 읽습니다. scripts 폴더에는 스킬 실행 시 필요한 스크립트를 추가할 수 있습니다.

작성 권고사항은 이렇습니다. SKILL.md는 500줄을 넘기지 말라고 권고하고 있습니다. 간결하게 핵심 지침만 담고, 상세한 내용은 보조 문서로 분리하세요. SKILL.md에 모든 걸 넣으면 매번 컨텍스트를 차지하지만, 분리하면 필요할 때만 로드됩니다. CLAUDE.md와 같은 원리입니다.

SKILL.md 파일을 보시면, 상단에 YAML frontmatter가 있습니다. name 필드가 슬래시 명령어가 됩니다. 예를 들어 name을 weekly-report로 지정하면 `/weekly-report`로 호출할 수 있습니다. description은 Claude가 자동 호출 여부를 판단하는 기준입니다. 그리고 본문에는 Claude가 따를 지침을 마크다운으로 작성합니다.

## Slide 9: Skills — 생성 방법과 호출 방법 (약 1분)

스킬을 만드는 방법은 두 가지가 있습니다.

첫 번째는 직접 생성입니다. `.claude/skills/` 아래에 스킬 이름으로 디렉토리를 만들고, SKILL.md 파일을 직접 작성하면 됩니다. name과 description을 frontmatter로 지정하고, 본문에 지침을 적으면 끝입니다.

두 번째는 Skill Creator를 사용하는 방법입니다. 이건 자동으로 설치되어 있는 공식 플러그인입니다. "코드 리뷰 스킬 만들어줘"라고 말하면 알아서 Skill Creator가 실행되어 SKILL.md를 자동으로 생성해줍니다. 이 방법을 권장합니다.

호출은 두 가지 방식이 있습니다. 직접 슬래시 명령어로 `/weekly-report`라고 호출할 수도 있고, "이번 주 보고서 만들어줘"라고 자연어로 말해도 Claude가 description을 보고 알아서 매칭해서 호출합니다.

이런 식으로 /code-review로 코드 리뷰를, /commit으로 커밋 메시지 생성을, /tc-generator로 테스트 케이스 도출을 할 수 있습니다.

매번 같은 지시를 반복하지 않아도 됩니다.

## Slide 10: MCP — Claude에 외부 프로그램을 연결해주는 도구 (약 1분)

MCP는 Claude에 외부 프로그램을 연결해주는 도구입니다. 구조를 보시면, Claude Code가 MCP Client이고, MCP Server가 외부 프로그램을 호출하는 기능들을 모아둔 프로그램입니다. 예를 들어 jira-server라는 MCP Server를 통해 Jira에 접근하는 구조입니다.

MCP 서버를 등록할 때는 `claude mcp add` 명령어를 사용합니다. 이때 `--scope` 옵션으로 범위를 지정할 수 있는데요. local은 이 프로젝트에서 나만, project는 이 프로젝트에서 팀 전체가, user는 모든 프로젝트에서 사용할 수 있습니다. 기본값은 local입니다.

사용 가능한 MCP 목록은 smithery.ai/servers에서 확인할 수 있고, 연동 방법도 안내되어 있습니다.

## Slide 11: MCP — 주의사항 (약 1분)

MCP를 사용할 때 반드시 알아야 할 주의사항이 있습니다. `/mcp` 명령어로 현재 연결된 MCP 목록을 확인할 수 있는데요.

문제는 MCP가 대화를 시작하는 순간, 활성화된 모든 MCP가 제공하는 기능 목록과 사용법을 컨텍스트에 로드한다는 겁니다. 예를 들어 Notion MCP만 15,000토큰, Slack MCP가 8,000토큰 — MCP만으로 27,000토큰 이상을 차지할 수 있습니다. 안 쓰는 MCP가 켜져 있으면 컨텍스트를 잡아먹는 주범이 됩니다.

그래서 미사용 MCP는 항상 꺼두세요. 그리고 기존 MCP가 너무 많은 도구를 포함하고 있다면, 필요한 기능만 담은 커스텀 MCP를 직접 만들어 사용하는 것도 권장합니다.

## Slide 12: Subagents — 별도 컨텍스트에서 일하는 독립 작업자 (약 2분)

Subagents는 별도 컨텍스트에서 일하는 독립 작업자입니다.

왜 필요한지 예시로 보겠습니다. "Confluence에서 결제 서비스 정책 찾아줘"라고 직접 시키면, 탐색 과정 전부가 내 컨텍스트에 쌓입니다. 반면 Subagent에게 위임하면, 별도 컨텍스트에서 탐색하고 결과 요약만 반환합니다.

만드는 법도 간단합니다. .claude/agents 폴더에 마크다운 파일을 만들고, name, description, 모델을 지정하면 됩니다. 모델은 작업 성격에 맞게 선택하면 비용을 줄일 수 있습니다. 단순 검색은 Haiku, 코드 리뷰는 Sonnet, 복잡한 설계에만 Opus를 쓰면 됩니다.

그리고 독립적인 작업은 Subagent를 동시에 여러 개 실행할 수 있습니다. 순차로 하면 6분 걸릴 작업이, 병렬로 하면 2분이면 끝납니다.

---

## Slide 13: Agent Teams — 여러 Claude가 동시에 협업 (약 1분 30초)

Agent Teams는 한 단계 더 나아갑니다. 여러 Claude가 팀을 이뤄서 동시에 작업합니다.

예를 들어 "JIRA-1234 버그를 에이전트 팀을 구성해서 수정해줘"라고 하면, 분석 담당이 Jira 티켓을 확인하고 원인을 파악하고, 프론트 담당이 화면 쪽을 수정하고, 백엔드 담당이 API 쪽을 수정하고, 테스트 담당이 수정사항 기반으로 테스트를 작성합니다. 서로 발견한 내용을 공유하면서 진행합니다.

Subagents는 독립적인 작업을 각자 하고 결과만 반환하는 거고, Agent Teams는 서로 의존적인 작업을 소통하면서 진행하는 겁니다.

혼자 다 안 해도 됩니다. 팀을 꾸려서 시키면 됩니다.

---

## Slide 14: 기타 기능 & 자주 사용하는 명령어 (약 1분 30초)

Hooks는 특정 이벤트가 발생하면 자동으로 실행되는 스크립트입니다. 파일을 수정하면 prettier가 자동으로 돌아가서 포맷팅을 맞춰주고, .env 같은 보호 파일을 수정하려고 하면 아예 차단하고 경고 메시지를 띄웁니다.

Plugins는 Skills, Hooks, MCP 설정을 하나로 묶어서 배포하는 패키지입니다. Git repo에 올려두면 팀 전체가 같은 환경에서 작업할 수 있습니다.

/loop는 세션 안에서 프롬프트를 반복 실행하는 기능입니다.

오른쪽에 자주 사용하는 명령어도 정리해뒀으니 참고해주세요. /init으로 CLAUDE.md를 생성하고, /clear로 컨텍스트를 초기화하고, /compact로 압축할 수 있습니다.

## Slide 15: 컨텍스트 관리 원칙 (약 2분)

그래서 실무에서 지켜야 할 원칙을 정리했습니다.

첫째, CLAUDE.md는 200줄 이하로 유지합니다. 빌드 명령, 코딩 컨벤션 같은 핵심 규칙만 넣으세요.

둘째, SKILL.md는 500줄 이하로 유지합니다. 넘으면 별도 파일로 분리하고, SKILL.md에서 참조하는 방식으로 관리합니다.

셋째, 불필요한 정보는 삭제합니다. "혹시 몰라서" 넣은 내용이 성능을 깎습니다.

넷째, 스킬로 분리합니다. 모든 정보를 한 파일에 넣지 말고, 작업별로 나눠서 관리하세요.

다섯째, 미사용 MCP는 비활성화하세요. 앞서 봤듯이 안 쓰는 MCP도 컨텍스트를 차지합니다.

여섯째, 작업 중 진행사항은 문서로 저장하세요. 컨텍스트가 길어지면 /compact로 압축할 수 있습니다.

일곱째, 작업이 끝나면 /clear로 초기화하세요. 다음 작업에 영향이 없도록요.

여덟째, /doctor로 상태를 확인할 수 있습니다. MCP 토큰 사용량이나 Context Usage Warnings을 확인할 수 있습니다.

짧고 핵심적으로 쓸수록, AI가 더 정확하게 동작합니다.

## Slide 16: 데모 — 지라 티켓 하나로 PR까지 (약 5분)

자, 지금까지 설명드린 내용이 실제로 어떻게 동작하는지 보여드리겠습니다.

제가 실무에서 구축한 워크플로우인데요. 지라 티켓 하나를 전달하면, Claude가 관련 프로젝트를 탐색하고, 코드를 분석하고, 수정한 다음, PR까지 생성합니다.

(데모 영상 재생)

보신 것처럼, 사람이 한 건 지라 티켓을 전달한 것뿐이고, 나머지는 Claude가 알아서 처리했습니다.

## Slide 17: 무엇을 하고 싶은가? (약 2분)

요즘 AI 활용 방법에 대해 수많은 정보가 넘쳐나고 있습니다. 과부하가 걸릴 정도로 새로운 기능이 나오고, 새로운 활용 방법이 등장합니다. 속도가 너무 빨라 따라가기 힘들죠.

어떻게 활용해야 하는지 많은 고민들을 하고 계실 것 같습니다. 그런데 이것보다 중요한 건, 과연 내가 무엇을 하고 싶은가입니다.

방법은 해결하고 싶은 문제를 하나씩 분해하면서 자연스럽게 따라옵니다. 다들 무엇을 하고 싶으신가요? 귀찮은 게 무엇인가요? 하기 싫은 게 무엇인가요? 무작위로 쏟아내보시면, 자연스럽게 그 과정 속에서 AI를 활용해 다양한 작업을 자동화한 모습을 보실 수 있을 거라고 생각합니다.

감사합니다.
